# Findings — AgenticOS Foundations

## What we now know

1. **Control plane = reasoning/execution separation** (sources 7, 8). Production
   agent OSes decouple "what to do" (semantic reasoning) from "how to observe and
   act" (capability-controlled execution). Sensai-HUD already does this: `config/`
   is the declarative control plane; `bin/` + gated writes are execution.

2. **Loop engineering is the current layer** (sources 2, 3). The progression is
   prompt → context → harness → **loop**: you design the system that prompts the
   agent. Sensai-HUD's skills-as-loops (preamble → workflow → telemetry) fit this.

3. **Long-running agents need context bridges** (source 1). Split an **initializer**
   (first-run setup) from a **coding/worker** agent (incremental progress), and
   persist state in artifacts a fresh context can read fast (progress log + git
   history + a JSON checklist whose entries are guard-railed). Directly informs the
   daily/weekly summary design and `agents/operations.md` as the activity bridge.

4. **LLM Wiki beats RAG for durable knowledge** (source 4). Compile sources once
   into interlinked markdown and keep it current, instead of re-deriving on every
   query. This *is* the `summaries/` (raw→processed→outputs) + `research/index.md`
   design; validates keeping compiled pages as the editable knowledge layer.

5. **Recursive self-improvement = capture → propose → evaluate → retain** (sources
   5, 6). DGM and the self-evolving-agents survey show measurable gains from an
   evolutionary loop that keeps only improvements. Sensai-HUD adopts the *loop*
   but bounds it to **human-gated** adoption (no autonomous skill/config rewrite),
   matching the repo's safety posture.

## Confidence
- Claims 1–4: corroborated (each backed by ≥1 primary source; 1 & 4 by two).
- Claim 5: corroborated by two sources; the "human-gated" bound is a Sensai-HUD
  design decision, not a literature claim.
